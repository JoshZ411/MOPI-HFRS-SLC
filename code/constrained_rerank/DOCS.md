# Constrained Rerank Package Documentation

## Scope of This Document

This document is only about the code in `code/constrained_rerank/`.

It does not document the whole repository. It explains the post-training constrained reranking implementation that sits on top of the saved SGSL recommender checkpoint.

## What This Package Is Solving

The code in `code/constrained_rerank/` solves a very specific problem:

- Start from a baseline top-K recommendation list generated by the trained recommender.
- Improve secondary properties of that list, mainly health alignment and list utility.
- Do not allow those improvements to freely destroy the original ranking quality.
- Make only bounded, interpretable edits.

This package is therefore a controlled inference-time editor for recommendation lists.

## Important Clarification

Despite the branch name and your shorthand, this folder is not implementing reinforcement learning in the standard sense.

There is:

- no policy network
- no reward optimization loop
- no trajectory rollout
- no value function
- no actor-critic or Q-learning update

What it does implement is a deterministic constrained reranking policy:

- generate a baseline anchor list
- scan a candidate pool
- greedily choose higher-health candidates
- enforce hard constraints before each swap
- accept or reject the final reranked output based on evaluation gates

If you refer to this folder as the RL implementation, the closest practical interpretation is that it behaves like a hand-coded policy over ranking lists, not a trained RL agent.

## High-Level Architecture

The folder contains six logical parts:

1. Entry and orchestration
2. Baseline anchor generation
3. Feasibility constraints
4. Greedy reranking policy
5. Evaluation and acceptance gates
6. Result logging

Those correspond directly to these files:

- `main.py`
- `anchor.py`
- `constraints.py`
- `reranker.py`
- `evaluation.py`
- `logging_utils.py`

There is also `__init__.py`, which only marks the directory as a package.

## End-to-End Control Flow

The package executes in this order:

1. Parse CLI configuration.
2. Load the saved SGSL checkpoint and graph.
3. Rebuild the frozen SGSL model.
4. Pick either the validation or test edge split.
5. Recompute user and item embeddings for that split.
6. Build a masked score matrix.
7. Extract:
   - an anchor top-K list
   - a larger top-M candidate pool
8. Evaluate the anchor list.
9. For each user, try to improve editable positions with better health-overlap candidates.
10. Reject any proposed swap that violates a hard rule.
11. Evaluate the reranked output.
12. Decide whether the reranked output passes the acceptance gates.
13. Save configuration, metrics, diagnostics, and comparison tables.

In short, this folder turns frozen embeddings into a baseline list, modifies that list conservatively, and then checks whether the modification was acceptable.

---

## Design Concept

This package depends on three ideas.

### 1. Anchor-first reranking

The baseline recommendation list is treated as the safe reference point. Every change is measured relative to that anchor.

### 2. Local list edits instead of retraining

The package never changes model weights. It only swaps items inside the ranked output.

### 3. Hard constraints plus final gates

There are two layers of safety:

- per-swap feasibility constraints
- experiment-level acceptance gates

That structure is why this package is useful: it gives a controllable way to bias recommendations toward health without reopening training.

## `code/constrained_rerank/main.py`

This is the package entrypoint. It is responsible for turning command-line settings into a complete reranking run.

### What this file does overall

It wires every other module together:

- `anchor.py` for model reconstruction and baseline list generation
- `constraints.py` for swap feasibility rules
- `reranker.py` for the actual list-editing logic
- `evaluation.py` for metrics and acceptance gates
- `logging_utils.py` for saving outputs

### `parse_args()`

Purpose:
Defines the full CLI surface for the reranker.

Arguments exposed here control four areas:

- file paths
- device selection
- ranking and candidate-pool sizes
- constraint and gate thresholds

The key arguments are:

- `--output_dir`: where results are written
- `--checkpoint_path`: which trained recommender checkpoint to load
- `--graph_path`: graph data needed to reconstruct SGSL
- `--K`: list length to optimize
- `--M`: candidate pool size to search over
- `--anchor_lock_positions`: number of leading positions that cannot change
- `--anchor_epsilon`: tolerated score drop for a replacement item
- `--max_swaps_per_list`: max number of accepted edits per user list
- `--ndcg_floor`: max allowed relative NDCG degradation
- `--split`: whether to operate on validation or test edges

Why this matters:
This function defines the operating policy of the reranker. Every experiment is just a different configuration of these controls.

### `resolve_device(device_str)`

Purpose:
Maps the CLI device choice into a real PyTorch device.

How it works:

- `auto` chooses CUDA if available, otherwise CPU
- `cuda` forces CUDA and raises if unavailable
- `cpu` forces CPU

Why it matters:
It prevents the rest of the pipeline from needing device-specific branching.

### `run(args)`

Purpose:
Executes the full constrained reranking pipeline.

How it works, step by step:

1. Prints configuration summary.
2. Resolves device and seed.
3. Saves the run configuration.
4. Optionally initializes W&B offline logging.
5. Loads the model and checkpoint with `load_model_and_data()`.
6. Chooses the split tensors from the checkpoint.
7. Extracts tags, features, and metadata needed downstream.
8. Generates embeddings with `get_embeddings_for_split()`.
9. Builds anchor and candidate lists with `get_anchor_list_and_scores()`.
10. Evaluates the anchor list with `evaluate_topk_list()`.
11. Constructs a `FeasibilityChecker`.
12. Applies the greedy editing policy with `constrained_rerank()`.
13. Aggregates diagnostics.
14. Evaluates the reranked result.
15. Checks hard and secondary gates with `compute_acceptance_gates()`.
16. Prints a formatted comparison table.
17. Saves structured outputs to disk.
18. Logs metrics if W&B is active.
19. Appends a repo-level phase log.

Why this matters:
`run()` is the only place where the package behaves like a full system instead of a group of helpers.

Operationally, this is the package's control tower.

## `code/constrained_rerank/anchor.py`

This file defines how the reranker reconstructs baseline recommendations from the frozen model.

The anchor list is critical because the rest of the package measures every change relative to it.

### `load_model_and_data(checkpoint_path, graph_path, device)`

Purpose:
Restore the trained SGSL recommender and all saved tensors needed for reranking.

How it works:

- loads the checkpoint from disk
- loads the graph from disk
- rebuilds an `SGSL` instance using saved hyperparameters
- loads the saved `state_dict`
- moves the model to the target device
- switches the model into eval mode

Inputs:

- checkpoint path
- graph path
- device

Outputs:

- reconstructed model
- raw checkpoint dictionary

Why it matters:
Without this function, the reranker would have no trustworthy baseline model to start from.

### `get_embeddings_for_split(model, feature_dict, edge_index, pos_edge_index, neg_edge_index, device)`

Purpose:
Recompute user and item embeddings for the requested split.

How it works:

- moves input tensors to the requested device
- runs `model.forward(...)` under `torch.no_grad()`
- returns only the final user and item embeddings

Why it matters:
This makes reranking operate on the same learned representation space as baseline evaluation.

Conceptually, this is the bridge from frozen model to list-editing stage.

### `get_anchor_list_and_scores(users_emb, items_emb, K, M, exclude_edge_indices)`

Purpose:
Create the safe baseline ranking and the larger search space used for edits.

How it works:

1. Compute the full user-item score matrix with a dot product.
2. Apply exclusion masking so blocked interactions are never recommended.
3. Extract the top-K highest-scoring items per user as the anchor list.
4. Extract the top-M highest-scoring items per user as the candidate pool.

Outputs:

- `anchor_topk`
- `anchor_scores`
- `candidate_pool`
- `candidate_scores`
- masked rating matrix

Why it matters:
This function defines the search space of the reranker.

The reranker is not free to inspect all items equally. It searches within the high-scoring candidate pool built here.

## `code/constrained_rerank/constraints.py`

This file defines the hard rules that govern which edits are legal.

It is the package's safety layer.

### `FeasibilityChecker`

Purpose:
Encapsulate the per-swap legality test.

Constructor parameters:

- `lock_positions`
- `epsilon`
- `max_swaps`

These become the rules used during reranking.

### `FeasibilityChecker.check_swap(...)`

Purpose:
Determine whether one proposed replacement can be accepted.

Inputs:

- the position being edited
- the anchor score at that position
- the proposed candidate score
- the current partially edited list
- the candidate item id
- the current swap count

Decision rules applied in order:

1. `position_lock`
   Reject if the position is inside the locked prefix.

2. `score_margin`
   Reject if the candidate score is less than `anchor_score - epsilon`.

3. `duplicate`
   Reject if the candidate already appears in the current list.

4. `budget_exceeded`
   Reject if the list has already used all allowed swaps.

Outputs:

- `True/False`
- rejection reason or `None`

Why it matters:
This single method is the core guardrail of the whole package.

### `ListDiagnostics`

Purpose:
Track what happened during reranking for one user list.

State tracked:

- number of attempted swaps
- number of accepted swaps
- number of rejections by each rule
- number of forced fallbacks to the anchor

### `ListDiagnostics.record_attempt(is_feasible, rejection_reason)`

Purpose:
Update counters after each attempted swap.

Why it matters:
This creates the diagnostic data used later to understand whether reranking succeeded because the policy was good or because constraints were loose.

### `ListDiagnostics.to_dict()`

Purpose:
Expose the per-list diagnostics in serializable form.

### `aggregate_diagnostics(all_diagnostics)`

Purpose:
Combine all per-user diagnostics into one summary.

Outputs include:

- total attempts
- total accepted swaps
- total rejections by each reason
- swap rate
- rejection rate
- forced-anchor rate

Why it matters:
This is how the package summarizes the behavior of the reranking policy over the whole evaluation population.

## `code/constrained_rerank/reranker.py`

This file contains the actual decision policy that edits ranked lists.

If `constraints.py` defines what is allowed, `reranker.py` defines what the package tries to do within those limits.

### `compute_health_overlap(user_tag, food_tags_candidates)`

Purpose:
Measure how well each candidate food matches the user's health tags.

How it works:

- broadcast the user tag vector across all candidate foods
- compute logical intersection
- compute logical union
- return Jaccard-style overlap scores

Why it matters:
This function is the package's utility heuristic.

The reranker does not learn a reward. It uses this explicit overlap score to decide which candidates look better for health.

### `constrained_rerank(anchor_topk, anchor_scores, candidate_pool, candidate_scores, checker, user_tags, food_tags, users=None)`

Purpose:
Apply the greedy constrained editing policy to one or more users.

This is the core algorithm in the folder.

How it works for each user:

1. Copy the anchor list into a mutable `current_list`.
2. Reset the swap counter.
3. Compute health-overlap scores for the entire candidate pool.
4. Sort the candidate pool from best health match to worst.
5. Iterate through positions in the top-K list.
6. Skip any position inside the locked prefix.
7. For each editable position, inspect candidates in descending health order.
8. Ignore candidates that are identical to the current anchor item.
9. Ignore candidates that do not improve health overlap relative to the current item.
10. Call `checker.check_swap(...)`.
11. If the candidate is feasible, insert it and increment the swap counter.
12. If no candidate passes, keep the original item at that position.

Outputs:

- reranked top-K tensor
- per-user diagnostics list

What this function is optimizing in practice:

- maximize health-overlap improvements first
- while respecting score quality and edit-budget constraints

Why it matters:
This function is the actual reranking policy.

It is greedy, local, and interpretable. It does not search globally for the best permutation, and it does not retrain anything.

This makes it easier to analyze and safer to deploy, but also means it can miss globally better list configurations.

## `code/constrained_rerank/evaluation.py`

This file evaluates baseline and reranked recommendation lists after they already exist.

It separates list evaluation from model inference.

### `evaluate_topk_list(top_K_items, edge_index, user_tags, food_tags, K, num_foods)`

Purpose:
Compute the full metric bundle for a precomputed top-K recommendation tensor.

Metrics returned:

- recall
- precision
- ndcg
- health_score
- avg_health_tags_ratio
- percentage_recommended_foods

How it works:

- determines which users are in the evaluated split
- builds ground-truth positive item sets from the edge list
- checks which recommended items are relevant
- computes ranking metrics
- computes health and coverage metrics

Why it matters:
This function lets the package compare anchor and reranked lists on equal terms.

### `compute_acceptance_gates(baseline_metrics, reranked_metrics, ndcg_floor=0.07)`

Purpose:
Decide whether the reranked result is acceptable.

This function implements the package's final experiment-level policy.

It computes two gate types.

#### Hard gate

Definition:
The relative NDCG drop must stay below the configured threshold.

Formula:

- if baseline NDCG is positive:
  `ndcg_drop_fraction = (baseline_ndcg - reranked_ndcg) / baseline_ndcg`
- otherwise the drop fraction is set to zero

Pass condition:

- `ndcg_drop_fraction <= ndcg_floor`

#### Secondary gate

Definition:
The secondary metrics must not get worse.

The tracked metrics are:

- `health_score`
- `avg_health_tags_ratio`
- `percentage_recommended_foods`

Pass condition:

- each of those deltas must be non-negative

Why it matters:
The package does not assume that any reranking is an improvement. It explicitly checks whether the edit policy stayed within acceptable tradeoffs.

### `generate_comparison_table(baseline_metrics, reranked_metrics, diagnostics_agg)`

Purpose:
Produce a readable text report of metrics and diagnostics.

What it includes:

- baseline values
- reranked values
- metric deltas
- aggregated reranker diagnostics

Why it matters:
This is mainly for fast human inspection of experiment outcomes.

## The Actual Policy Implemented by This Folder

If you compress the whole folder into one behavioral description, it is this:

1. Build the strongest baseline list available from the trained model.
2. Look at a slightly larger pool of high-scoring candidates.
3. Prefer candidates with better health-tag overlap.
4. Do not edit protected top positions.
5. Do not insert an item that is too far below the baseline score.
6. Do not duplicate items.
7. Do not exceed the per-list swap budget.
8. After editing, reject the overall result if ranking quality falls too far.

That is the package's policy in operational form.

---

## What Each File Contributes to That Policy

- `main.py` decides how the full reranking experiment is run.
- `anchor.py` defines the baseline list and candidate search space.
- `constraints.py` defines what edits are legal.
- `reranker.py` defines which legal edits are attempted.
- `evaluation.py` defines how success and failure are measured.
- `logging_utils.py` defines how results are stored.

This division is clean and deliberate. It makes the package easier to change because each file owns one responsibility.

---

## Practical Strengths of This Implementation

- It is interpretable. Every swap can be explained.
- It is modular. The reranking layer is separate from model training.
- It is safer than unconstrained reranking because it enforces hard per-swap rules.
- It is easy to experiment with because most behavior is controlled by CLI thresholds.

## Practical Limitations of This Implementation

- It is greedy, so it does not solve a global optimization problem over the full list.
- It uses a handcrafted health-overlap heuristic instead of a learned reranking objective.
- It depends on candidate pool quality; if a good replacement is not in top-M, it cannot be chosen.
- It is not a reinforcement learning system, so it does not learn a reranking policy from feedback.

## Summary

The code in `code/constrained_rerank/` implements a deterministic, post-training constrained reranker for food recommendations.

Its logic is:

- reconstruct the frozen recommender
- create an anchor top-K list and candidate pool
- greedily propose health-improving swaps
- enforce hard feasibility constraints
- evaluate the edited lists
- accept or reject the result based on ranking and secondary-metric gates

So if you want the simplest accurate description of this folder, it is not an RL training module. It is a constrained inference-time ranking editor with explicit rules, explicit diagnostics, and explicit acceptance criteria.
